# CLAUDE.md

Web portfolio for **Hilmi Zikri** ("Zikri"). Next.js 16 (App Router) + TypeScript + react-three-fiber/drei, migrated from a static HTML/CSS/JS site.

## Stack
- Next.js 16, React 19, TypeScript
- GSAP + ScrollTrigger for the Projects page 3D sticky-cards effect (`components/animations/3d-sticky-cards/`)
- Hand-written CSS (no Tailwind/CSS Modules) site-wide, imported once via `styles/main.css` in `app/layout.tsx`
- EmailJS (CDN script, no npm dep) for contact form
- react-three-fiber / drei / three / simple-icons — used by `FrozenKeyboard.tsx` (the 3D keyboard in the landing page's Tech Stack section)

## Commands
- `npm run dev` — dev server
- `npm run build` — production build (also type-checks)
- `npm run lint` — eslint

## Structure
- `app/` — routes: `/` (landing), `/projects`, `/about`, `/contact`. Each `page.tsx` just renders a component from `components/`.
- `components/` — one folder per page (`Landing/` — `HeroIntro.tsx`, `RolePicker.tsx`, `TechStackSection.tsx`, see below; `Projects/`, `About/`, `Contact/`) plus shared `Nav.tsx`, `Footer.tsx`; `components/animations/<source-name>/` for ported GitHub animation effects (e.g. `3d-sticky-cards/`); `components/ThreeDPortfolio/` — only `FrozenKeyboard.tsx` + its `LanguageProvider.tsx`/`SeasonProvider.tsx` dependencies remain (see Important context — the rest of this folder was a ported 3D portfolio page, deleted).
- `lib/` — typed data, e.g. `projects.ts` (Projects page sticky-card content); `lib/threeDPortfolio/` — i18n/seasons/skills/useIsMobile helpers, still load-bearing for `FrozenKeyboard` and imported directly by the landing page's Tech Stack section.
- `styles/` — `main.css` chains `@import`s: tokens → base → typography → components/* → pages/* → utilities/* → responsive.
- `public/Assets/` — images. `public/CNAME` — GitHub Pages custom domain. `public/3d-portfolio/sounds/` — key-press audio for the landing page's Tech Stack keyboard (`fonts/` subfolder is an unused leftover, not referenced anywhere).

### Landing page (`/`) — 3 stacked sections + Footer
`app/page.tsx` renders `<main><HeroIntro/><RolePicker/><TechStackSection/></main><Footer/>`. Each section is a normal `min-height: 100dvh` flex-centered block in `styles/pages/landing.css` — earlier attempts used `position: fixed`/`vh`-math overlays (a leftover from the original static site) and that caused everything to visually collide once real content existed below the hero; don't reintroduce fixed/vh positioning here.
- `HeroIntro.tsx` — Section 1, name reveal: typewriter "Hi! I'm" + big `<h1>ZIKRI</h1>` reveal, `no-scroll` body-lock until the reveal transition finishes. Root carries `className="landing"` deliberately — `styles/utilities/animations.css` keys off `.landing`'s presence to suppress the generic page-fade-in on the home page.
- `RolePicker.tsx` — Section 2, drag-and-drop role picker ("Hi! I'm a [Drop Here]" + draggable "Full-Stack Dev" pill). Dropping the pill adds `body.fade-out` (0.5s opacity transition, `styles/base.css`) then hard-navigates to `/projects` via `window.location.href` (was `/projects#programming-page` — that anchor was a leftover from the deleted accordion Projects page and never existed in the current sticky-cards version; fragment dropped). **This fade-to-blank-then-navigate is intentional, not a bug** — confirmed by driving a real Chromium instance with Playwright (see Important context) end-to-end: the drop lands correctly and the redirect fires with zero console errors. It reads as "the page froze" to a first-time user because the whole page goes blank for 0.5s right before the URL changes — if this UX gets reported as broken again, it's this transition, not the drag/drop logic. Near-verbatim port of the original vanilla drag/drop handlers, just scoped to this component's own root instead of the old shared-fixed-overlay version.
- `TechStackSection.tsx` — Section 3, reuses `FrozenKeyboard` from `components/ThreeDPortfolio/FrozenKeyboard.tsx` directly — **no** `LanguageProvider`/`SeasonProvider`/`FrozenBackground` wrapping; `useLanguage()`/`useSeason()` both degrade gracefully to defaults without a provider (confirmed intentional, not a workaround). `.tech-stack-section` is a 200dvh scroll runway; `.tech-stack-sticky` (`position: sticky; top: var(--nav-height)`) pins the canvas at viewport-minus-navbar and also carries `data-kb-section="stack"` — that attribute (not the outer 200dvh section) is what `FrozenKeyboard`'s internal `IntersectionObserver` watches, so the hero→stack pose lerp animation is in sync with when the canvas is actually visible, not firing while it's still off-screen below the fold.
  - `FrozenKeyboard.tsx` was adapted to support standalone reuse: `SECTION_STATES.stack`'s `posY` was raised from the source's `-0.6` to `-0.25` (keeps the model closer to the heading above it — the visual gap was the 3D camera framing, not CSS spacing); the in-canvas `Text3D`/`Text` hover callout (extruded 3D font, tilted) was removed and replaced with an `onHoveredIconChange?: (icon: SkillIcon | null) => void` callback prop, wired to `TechStackSection.tsx`'s own plain-CSS DOM callout. This is now `FrozenKeyboard`'s only consumer — it used to be dual-consumed with the now-deleted `/3d-preview` page's `ThreeDPortfolioPage.tsx`, so this callback prop and the standalone-reuse adaptations are no longer strictly required by two call sites, just by this one.
- `--nav-height: clamp(60px, 10vh, 80px)` token (`styles/tokens.css`) exists specifically so in-page `position: sticky` content (like `.tech-stack-sticky`) can clear the also-sticky `<nav>` instead of pinning underneath it (nav's `z-index: var(--z-nav)` wins the overlap otherwise). Reuse this token for any future sticky-within-page-content, don't hardcode the nav height again.
- `HeroIntro.tsx`'s mount effect adds `document.body.classList.add('no-scroll')` during the reveal animation, removed once a `transitionend` fires. Since `<Nav/>` is always mounted independent of the current page, a nav click can unmount `HeroIntro` mid-animation — the effect cleanup unconditionally removes `no-scroll` and cancels pending timeouts/listeners (a `cancelled` flag) specifically to prevent that class getting stuck on `<body>` forever (which freezes scroll site-wide until reload). Any future component doing a similar "lock scroll until an animation finishes" pattern needs the same unconditional-cleanup treatment.
